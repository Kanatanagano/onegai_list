# Recursion チーム開発の詳細(Golang webAPI)

## 概要
Recursionでは、学生が実践的なチーム開発を通じてスキルを習得し、企業で即戦力として活躍できるようにします。

本ドキュメントでは、実際のチーム開発での開発フロー、チームのやりとり、期間、成果物、レベル、技術選定、メンターの役割、Git管理方法、オンラインでのやり方、タスク管理、要件定義と開発の自由度、そして取得したスキルセットについて詳しく説明します。

## 開発フロー

1. **要件定義**
   - チームリーダーを中心にコアタイムと製作物の決定
   - 期間内に達成するべき目標を明確化
   - 外部Web APIについての情報収集と理解
   - 各メンバーがGolangを使った小規模なデモアプリケーションを作成

2. **技術選定**
   - フロントエンド: HTML/CSS
   - バックエンド: Golang (Ginフレームワーク)
   - データベース: MySQL

3. **設計**
   - 設計方針をチーム全体で共有
   - HTTPリクエストとレスポンスの形式を確認
   - フォルダ構成、コード規約をチームに周知

4. **開発**
   - タスクを細分化し、各メンバーが担当部分を実装
   - コミットメッセージは第三者にもわかりやすいように記述
   - プルリクエストを通じてコードレビューを実施

5. **レビューと改善**
   - 開発終了後の振り返り
   - プロジェクト全体のフィードバックを実施し、改善点を特定

## 説明

チーム開発では、プログラミング初心者から中級者まで様々なバックグラウンドを持つメンバーが参加します。今回のGolang Web APIプロジェクトでは、メンバー全員がGolang未経験者であったため、以下のような段階的な学習プロセスを取り入れました。

### 学習プロセス

1. **基礎の習得**
   - 各メンバーが簡単なリクエスト・レスポンスを処理するウェブサイトを作成し、Golangの基礎を学びました。

2. **応用スキルの習得**
   - JSON形式でのデータの送受信を行うウェブサイトを作成し、より実践的なスキルを身につけました。

3. **フレームワークの導入**
   - 開発生産性を向上させるために、HTTPメソッドに沿ったコードを書きやすいGinフレームワークを導入し、GolangからGinを用いた開発に移行しました。

### 開発の進行

このように、チームのスキルレベルに合わせて段階的に難易度を上げながら学習し、最終的な成果物を作成しました。具体的な開発フローに沿ってプロジェクトを進行することで、チーム全員が効率的に学習し、実践的な開発スキルを習得することができました。

これにより、以下のような成果が得られました：
- **迅速なスキルアップ**: 段階的な学習プロセスにより、全員が基礎から応用までしっかりと理解できました。
- **効率的な開発**: Ginフレームワークの導入により、コードの可読性とメンテナンス性が向上しました。
- **チームの協力**: リアルタイムコミュニケーションツールやGitHub Issuesを活用することで、スムーズなタスク管理と問題解決が可能になりました。

このプロジェクトを通じて、メンバー全員が実践的なスキルを身につけ、将来の開発に役立つ貴重な経験を積むことができました。

## 説明
### チームのやりとり

ミーティングでは、メンバーが担当している機能についての挙動を確認し、進捗を共有しました。その後、追加機能のディスカッションを行い、各メンバーの担当部分を決定しました。

私たちのチームでは、コアタイムを導入しており、その時間帯にOvice内で全員が集まって共同開発を進めています。これにより、リアルタイムでのコミュニケーションが可能になり、開発の効率が向上しました。

### 効果と成果

これらのツールを利用することで、以下のメリットが得られました：
- **迅速な問題解決**: リアルタイムのコミュニケーションにより、問題が発生した際にすぐに対処できるようになりました。
- **効率的なタスク管理**: GitHub IssuesやDiscordを活用することで、タスクの進捗状況や質問の管理が一元化され、開発がスムーズに進みました。
- **開発速度の向上**: コアタイム中に集中して開発を進めることで、全体の開発速度が大幅に向上しました。

これらのコミュニケーションツールと手法を導入することで、チーム全体の連携が強化され、プロジェクトの成功に繋がりました。

### 具体的な説明

#### フロントエンド
HTML/CSSを使用して、ユーザーインターフェースを構築します。HTMLはウェブページの基本構造を定義し、CSSでスタイルを適用して見た目を整えます。例えば、ユーザーがレシピを検索するときに入力内容をリアルタイムでサーバーに送信し、結果を表示する機能を実装します。

#### バックエンド
Golangを使ってWeb APIを構築します。Ginフレームワークを用いることで、ルーティングやミドルウェアの設定が簡単になり、効率的にAPIを開発できます。バックエンドでは、ユーザー認証、レシピデータの管理、検索機能の実装を行います。高い並行処理性能を持つGolangを使うことで、スケーラブルなAPIを実現します。

#### データベース
MySQLを使用して、ユーザー情報を管理します。リレーショナルデータベースであるMySQLは、データの整合性を保ちつつ、高速な検索クエリを実行できるため、レシピ検索機能に最適です。SQLクエリを用いてユーザー認証を行います。

このプロジェクトを通じて、フロントエンドとバックエンドの開発、データベースの設計と操作といったフルスタック開発の実践的なスキルを習得できます。

## Git管理方法

### ブランチ戦略

私たちのチームでは、効率的なGit管理方法として、`main`ブランチと`develop`ブランチの2つのブランチを使用しています。

- **`main`ブランチ**: このブランチは、常に安定したリリース可能なコードを保持します。完成した機能や修正は、十分なテストを経てから`main`ブランチにマージされます。
- **`develop`ブランチ**: 開発中のコードはこのブランチに集約されます。新機能や修正はまず`develop`ブランチにマージされ、テストとレビューを経て安定した状態になった後、`main`ブランチにマージされます。

### プルリクエスト

私たちのチームでは、全ての変更をプルリクエストを通じて行います。プルリクエストのプロセスは以下の通りです。

1. **プルリクエストの作成**: 各メンバーは、作業が完了したら、自分の作業ブランチから`develop`ブランチに向けてプルリクエストを作成します。
2. **コードレビュー**: プルリクエストが作成されたら、チームのメンバーおよびメンターがコードレビューを行います。レビューでは、コードの品質、スタイル、一貫性をチェックし、必要に応じてフィードバックを提供します。
3. **フィードバックの反映**: レビュワーからのフィードバックを受けた後、プルリクエストの作成者は修正を行い、再度レビューを依頼します。
4. **マージ**: 全てのレビューが完了し、コードが承認されたら、プルリクエストを`develop`ブランチにマージします。十分なテストが完了し、`develop`ブランチが安定した状態になったら、`main`ブランチにマージされます。

### Git Rebaseの活用

私たちのチームでは、Git Rebaseを積極的に活用しています。Rebaseを使用することで、以下のようなメリットを享受しています。

- **履歴の整然化**: Rebaseを使用することで、コミット履歴を整理し、一本化された綺麗な履歴を保つことができます。
- **コンフリクトの解消**: 他のブランチの変更を取り込む際にRebaseを使うことで、コンフリクトの解消が容易になり、最新の状態を反映させながら作業を進められます。
- **コードレビューの効率化**: 整然とした履歴を持つことで、プルリクエストのレビューがスムーズに行われ、変更内容の把握が容易になります。

### メリット

- **明確なブランチ戦略**: `main`ブランチと`develop`ブランチを明確に分けることで、安定したコードと開発中のコードを分離し、混乱を防ぎます。
- **コード品質の向上**: プルリクエストとコードレビューのプロセスにより、コードの品質が向上します。レビューによりバグや改善点を早期に発見できます。
- **チームの協力**: チームメンバー全員がプルリクエストを通じてプロジェクトの進捗を把握でき、共同作業が円滑に行われます。
- **履歴の整然化**: Git Rebaseを活用することで、コミット履歴を綺麗に保ち、過去の変更を追跡しやすくします。

このようなGit管理方法を導入することで、プロジェクトの品質と開発効率を高めることができます。

## 要件定義と開発の自由度

### 要件定義

プロジェクトの成功には明確な要件定義が欠かせません。私たちのチームでは、以下のプロセスを通じて要件を定義しています。

- **詳細なヒアリング**: プロジェクト開始時にステークホルダーと詳細なヒアリングを行い、ニーズや期待を明確に理解します。この段階で、プロジェクトの目標や主要な機能、ユーザーの要件を具体的に把握します。
- **ユーザーストーリーの具体化**: ヒアリング内容を基に、ユーザーストーリーを作成します。ユーザーストーリーは、ユーザーの視点から見た機能や操作フローを具体的に記述したもので、これにより開発チーム全体がプロジェクトの全体像を共有しやすくなります。

### 開発の自由度

私たちのチームでは、各メンバーが主体的に取り組むことができるよう、開発の自由度を重視しています。

- **技術選定や設計における意見交換**: 各チームメンバーが技術選定や設計について意見を出し合います。チーム内での議論を通じて、最適な技術スタックや設計方針を決定します。このプロセスにより、メンバー全員が納得し、モチベーション高くプロジェクトに取り組むことができます。
- **自由な発想での開発を奨励**: 各メンバーが自身の創造力を発揮できる環境を整えています。新しいアイデアや改善提案は積極的に取り入れ、柔軟に対応します。これにより、チーム全体が持つ技術力と創造力を最大限に引き出し、革新的な解決策や機能を実現します。

このように、明確な要件定義と高い開発の自由度を両立させることで、プロジェクトの成功とチームの成長を促進しています。
